# 跨平台自动化工作流与系统服务配置实战指北

## 概述
本教程记录了如何将日常繁琐的开发环境启动流程实现全自动化。主要涉及：
1. **Ubuntu**：后台服务的 Systemd 守护进程托管、NVM 环境平滑升级及常用运维指令。
2. **Windows**：开机自动免密进入桌面，同时保留系统锁屏密码。
3. **macOS**：底层定时开机配置、带重试与延时机制的开机自启脚本、系统级权限踩坑记录，以及快捷部署配置的命令集。

---

## 1. Ubuntu 环境配置：Systemd 托管与 NVM 升级

### 1.1 常规程序的 Systemd 服务配置 (以 Qwenpaw 为例)
将占用终端的前台应用转为后台服务，实现开机自启与崩溃重启。

**1. 编写服务配置文件**
`sudo nano /etc/systemd/system/qwenpaw.service`
```ini
[Unit]
Description=Qwenpaw App Service
After=network.target

[Service]
User=ai
# 指定工作目录以确保日志文件写入正确位置
WorkingDirectory=/home/ai/qwenpaw
ExecStart=/home/ai/.qwenpaw/bin/qwenpaw app
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

**2. 加载与启动**
```bash
sudo systemctl daemon-reload
sudo systemctl enable qwenpaw
sudo systemctl start qwenpaw
```

### 1.2 自带后台机制程序的配置 (以 Hermes-Web-UI 为例)
对于执行后会 `fork` 子进程并自动退回终端的程序（如 Node.js 服务），必须在配置中增加 `Type=forking`。

`sudo nano /etc/systemd/system/hermes-web-ui.service`
```ini
[Unit]
Description=Hermes Web UI Service
After=network.target

[Service]
User=ai
WorkingDirectory=/home/ai/hermes_web_ui
Type=forking
ExecStart=/home/ai/.npm-global/bin/hermes-web-ui start
ExecStop=/home/ai/.npm-global/bin/hermes-web-ui stop
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

### 1.3 NVM 环境升级与全局包无缝迁移
在升级 Node.js 环境时，由于历史遗留的 npm `prefix` 配置，会导致 NVM 报错。处理流程如下：

**1. 清除旧配置与冲突目录**
```bash
# 强制使用新版并删除冲突的 prefix 配置
nvm use --delete-prefix v24.17.0
npm config delete prefix

# 删除旧的全局包目录，避免环境变量污染
rm -rf ~/.npm-global
```

**2. 初始化新环境**
```bash
nvm alias default 24
npm config set registry https://registry.npmmirror.com
npm install -g hermes-web-ui pnpm @anthropic-ai/claude-code
```
*注：环境切换后，全局包的绝对路径会发生改变，需使用 `which` 命令重新获取路径，并更新对应的 Systemd 配置文件，随后执行 `daemon-reload`。*

### 1.4 Systemd 常用运维指令备忘
应用被 Systemd 接管后，严禁再使用应用原生的启动/停止命令（如 `npm run start` 等），应统一使用系统服务命令：

```bash
# 重载系统服务（修改 .service 文件后必须执行）
sudo systemctl daemon-reload

# 停止服务（在升级应用程序前必须先执行）
sudo systemctl stop <服务名>

# 启动服务
sudo systemctl start <服务名>

# 重启服务
sudo systemctl restart <服务名>

# 查看服务状态（按 q 退出查看）
sudo systemctl status <服务名>

# 设置开机自动启动 / 取消开机自动启动
sudo systemctl enable <服务名>
sudo systemctl disable <服务名>

# 查看服务完整日志（实时滚动监控输出）
sudo journalctl -u <服务名> -f
```

## 2. Windows 环境配置：自动登录与锁屏分离

**需求背景**：实现定时开机后自动登录进入桌面（以启动虚拟机和后台服务），但人离开时仍可通过 `Win + L` 锁屏保护隐私。

### 2.1 恢复被隐藏的免密登录选项
在 Windows 10/11 中，通过修改注册表恢复 `netplwiz` 中的复选框：
以管理员身份运行 CMD，执行：
```cmd
reg add "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\PasswordLess\Device" /v DevicePasswordLessBuildVersion /t REG_DWORD /d 0 /f
```

### 2.2 配置自动登录
1. 按 `Win + R` 运行 `netplwiz`。
2. 选中目标用户，**取消勾选**“要使用本计算机，用户必须输入用户名和密码”。
3. 点击“应用”，在弹出的对话框中精确输入该账户的系统登录密码并确认。
4. 重启生效。系统将在开机时自动登入桌面，锁屏功能及密码验证逻辑不受影响。

## 3. macOS 环境配置：定时开机与 SSH 自动化

**需求背景**：Windows 虚拟机在 07:00 启动，macOS 在 07:10 启动，随后 macOS 自动打通至虚拟机的 SSH 隧道。

### 3.1 设定系统级定时开机/唤醒
macOS 已移除电源调度的图形界面，需使用 `pmset` 命令行工具。

**开启定时开机脚本 (`autoboot_on.sh`)**
```bash
#!/bin/bash
# 设定每周一至周日 07:10 开机/唤醒
sudo pmset repeat wakeorpoweron MTWRFSU 07:10:00
pmset -g sched
```

**关闭定时开机脚本 (`autoboot_off.sh`)**
```bash
#!/bin/bash
sudo pmset repeat cancel
pmset -g sched
```

### 3.2 编写高可用 SSH 隧道连接脚本
此脚本具备 30 秒初始延时（等待网络握手）、多次轮询重试、系统原生通知及日志记录功能。

**`~/tools/sh/vm_connect.sh`**
```bash
#!/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin

echo "$(date): 脚本被触发，等待 30 秒让系统网络就绪..."
sleep 30

MAX_RETRIES=5
RETRY_INTERVAL=60

if pgrep -f "ssh -fN vm" > /dev/null; then
    echo "$(date): SSH 隧道已经在运行中。"
    osascript -e 'display notification "SSH 隧道已经在运行中。" with title "VM 连接"'
    exit 0
fi

echo "$(date): 准备开始连接..."
for ((i=1; i<=MAX_RETRIES; i++)); do
    echo "$(date): 尝试第 $i 次连接 vm..."
    
    ssh -o ConnectTimeout=10 -fN vm

    if [ $? -eq 0 ]; then
        echo "$(date): ✅ 连接 VM 成功！"
        osascript -e 'display notification "成功连接到 VM 并已建立端口转发！" with title "VM 连接"'
        exit 0
    else
        echo "$(date): ❌ 第 $i 次连接失败。"
        if [ $i -lt $MAX_RETRIES ]; then
            sleep $RETRY_INTERVAL
        fi
    fi
done

echo "$(date): ⚠️ 尝试 $MAX_RETRIES 次均失败，退出。"
exit 1
```

### 3.3 配置 LaunchAgent 实现开机自启

**排错关键点 (TCC 权限限制)**：
macOS 针对 `Desktop` (桌面) 和 `Documents` (文稿) 目录有严格的隐私限制。后台服务若读取这些路径内的脚本，将直接报错 `Operation not permitted`。
**解决方案**：将脚本统一存放在用户主目录下的非受限文件夹中（如 `~/tools/sh/`）。

**1. 创建配置文件 `~/tools/sh/com.alex.vmtunnel.plist`**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.alex.vmtunnel</string>
    <key>ProgramArguments</key>
    <array>
        <string>/bin/bash</string>
        <string>/Users/alex/tools/sh/vm_connect.sh</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>StandardOutPath</key>
    <string>/Users/alex/tools/sh/vmtunnel.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/alex/tools/sh/vmtunnel.log</string>
</dict>
</plist>
```

**2. 快捷部署与加载指令备忘**
维护或更新配置文件后，复制执行以下命令完成部署加载操作：

```bash
# 将配置文件拷贝至系统级用户 LaunchAgents 目录
cp ~/tools/sh/com.alex.vmtunnel.plist ~/Library/LaunchAgents/

# 卸载旧配置（尾部 2>/dev/null 用于静默忽略尚未加载时的报错）
launchctl unload ~/Library/LaunchAgents/com.alex.vmtunnel.plist 2>/dev/null

# 加载新配置（执行后将根据 RunAtLoad=true 规则立即触发）
launchctl load ~/Library/LaunchAgents/com.alex.vmtunnel.plist

# 打印并检查已部署的配置文件内容
cat ~/Library/LaunchAgents/com.alex.vmtunnel.plist
```

## 4. 附加技术探索：RustDesk 自动连接脚本 (归档)

**实现目标**：开机监控 RustDesk 进程，就绪后通过命令行参数自动发起远程桌面连接。
**技术难点及废弃原因**：虽然可以通过 `RustDesk --connect <ID>` 命令成功发起连接，且通过 `while` 轮询和超时控制实现了可靠的进程监控，但在实际场景中，RustDesk 客户端在冷启动后，获取节点与设备在线状态（握手中继服务器）通常存在不可控的网络延时（约 1 分钟）。硬编码延时无法保证连接的稳定性，故选择放弃全自动化，退回手动触发。

**备查代码 (`vm_rustdesk.sh`)**：
```bash
#!/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin

RD_BIN="/Applications/RustDesk.app/Contents/MacOS/RustDesk"
DEVICE_ID=8605810
LOG_FILE="$HOME/tools/sh/vm_rustdesk.log"
MAX_WAIT=120
ELAPSED=0

if [ ! -f "$RD_BIN" ]; then
    exit 1
fi

while ! pgrep -x "RustDesk" > /dev/null; do
    if [ $ELAPSED -ge $MAX_WAIT ]; then
        exit 1
    fi
    sleep 2
    ((ELAPSED+=2))
done

# 等待应用连接中继服务器获取设备状态
sleep 10
"$RD_BIN" --connect $DEVICE_ID &
exit 0
```
