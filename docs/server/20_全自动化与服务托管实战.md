## 1. Ubuntu 篇：使用 Systemd 托管服务与 NVM 环境升级

以前每次启动后端 API，都要在终端里手动挂着，终端一关服务就停，且无法开机自启。解决方案是使用 Linux 原生的 `Systemd` 进行托管。

### 1.1 将常规终端应用转为 Systemd 服务 (以 Qwenpaw 为例)

对于运行后一直霸占终端输出的程序，配置方法如下：

**1. 获取程序绝对路径**
```bash
which qwenpaw
# 假设输出：/home/ai/.qwenpaw/bin/qwenpaw
```

**2. 编写 Service 文件**
创建文件 `/etc/systemd/system/qwenpaw.service`：
```ini
[Unit]
Description=Qwenpaw App Service
After=network.target

[Service]
User=ai
# 指定工作目录，确保程序原本写在当前目录的 logs 还能正常输出
WorkingDirectory=/home/ai/qwenpaw
ExecStart=/home/ai/.qwenpaw/bin/qwenpaw app
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```

**3. 启用并启动服务**
```bash
sudo systemctl daemon-reload
sudo systemctl enable qwenpaw
sudo systemctl start qwenpaw
```
*(日后如需升级程序，遵循 `stop` -> 更新 -> `start` 的规范即可)*。

### 1.2 适配自带后台机制的应用 (以 Hermes-Web-UI 为例)

有些 Node.js 程序运行后会 `fork` 一个后台进程（打出 PID 后立刻回到终端）。对于这类程序，`Systemd` 的配置需要做微调。

**核心差异点：必须增加 `Type=forking`**

```ini
[Unit]
Description=Hermes Web UI Service
After=network.target

[Service]
User=ai
WorkingDirectory=/home/ai/hermes_web_ui

# 专门针对自带后台机制的程序
Type=forking

ExecStart=/home/ai/.npm-global/bin/hermes-web-ui start
ExecStop=/home/ai/.npm-global/bin/hermes-web-ui stop
Restart=always
RestartSec=3

[Install]
WantedBy=multi-user.target
```
> **⚠️ 避坑指南：** 
> 如果启动服务时报错 `Job failed ... exit code`，极大可能是因为你之前在终端手动运行的服务还没关，导致端口被占用。记得先通过原程序的 `stop` 命令或 `kill` 杀掉旧进程，再交由 `systemctl` 接管。

### 1.3 踩坑实录：NVM 升级 Node 环境与全局包迁移

在部署 Node.js 应用时遇到版本过低警告，决定彻底改用 `nvm` 管理 Node 版本，并配置淘宝源。

**1. 安装 NVM 并处理旧配置冲突**
由于早期为了免 sudo 全局安装 npm 包，曾经修改过 npm 的 `prefix`。这会导致 nvm 报错：*`incompatible with nvm...`*

**解决方案（清理旧配置）：**
```bash
# 1. 解除 prefix 冲突并使用新版本 (如 v24.17.0)
nvm use --delete-prefix v24.17.0
npm config delete prefix

# 2. 删除以前的旧全局目录（斩草除根）
rm -rf ~/.npm-global

# 3. 设置默认 Node 版本
nvm alias default 24
```

**2. 恢复全局工具与淘宝源**
在全新隔离的 Node 24 环境下重构生态：
```bash
npm config set registry https://registry.npmmirror.com
npm install -g hermes-web-ui pnpm @anthropic-ai/claude-code
```
*(最后别忘了用 `which hermes-web-ui` 获取新的 nvm 路径，去更新 Systemd 的 `ExecStart` 路径并 `daemon-reload`。)*

## 2. Windows 篇：开机自动进桌面 (保留锁屏密码)

**核心诉求：** Windows 虚拟机宿主机每天定时开机，希望它自动跳过密码直接进桌面，把里面的虚拟机和各种服务拉起来。但我又不能取消密码（因为需要局域网访问，且人走需要 `Win+L` 锁屏保护隐私）。

### 2.1 找回隐藏的 netplwiz 选项
在 Win10/11 中，运行 `netplwiz` 时，顶部的“要使用本计算机，用户必须输入用户名和密码”复选框通常被微软隐藏了。

**恢复方法：**
以管理员身份运行 CMD，输入以下命令修改注册表：
```cmd
reg add "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\PasswordLess\Device" /v DevicePasswordLessBuildVersion /t REG_DWORD /d 0 /f
```

### 2.2 配置自动免密登录
1. 按 `Win + R` 运行 `netplwiz`。
2. 选中当前用户名，**取消勾选**“要使用本计算机，用户必须输入用户名和密码”。
3. 点击“应用”，在弹出的对话框中**极其仔细地**输入两次开机密码（输错不会立刻报错，但会导致重启后卡锁屏）。
4. 完成！重启后将直达桌面。随手按下 `Win + L` 仍可锁屏。

## 3. macOS 篇：定时开机与带重试机制的自动化脚本

为了配合 Windows 的定时开机，我的 Mac 需要在开机后自动连接 SSH 隧道（暴露虚拟机端口）。由于 Windows 启动虚拟机需要时间，Mac 的脚本必须具备“失败重试”和“可视化弹窗”功能。

### 3.1 使用 pmset 配置定时开机 (无需图形界面)

现在的 macOS 移除了定时开关机的 UI，我们可以直接用底层命令实现。建立两个脚本方便切换：

**启用定时开机 (`enable_autoboot.sh`)：**
```bash
#!/bin/bash
echo "⏳ 正在设置每天早上 7:10 自动开机/唤醒..."
sudo pmset repeat wakeorpoweron MTWRFSU 07:10:00
pmset -g sched
```
*(注：设为 7:10 是为了给 7:00 开机的 Windows 留出充足的系统和 VM 启动时间。)*

**取消定时开机 (`disable_autoboot.sh`)：**
```bash
#!/bin/bash
echo "⏳ 正在取消自动开机/唤醒计划..."
sudo pmset repeat cancel
pmset -g sched
```

### 3.2 编写带原生通知的 SSH 自动重试脚本

创建 `~/connect_vm.sh`，利用 Mac 的 `osascript` 发送原生系统通知，利用 `for` 循环实现失败轮询：

```bash
#!/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin

MAX_RETRIES=5
RETRY_INTERVAL=60

if pgrep -f "ssh -fN vm" > /dev/null; then
    osascript -e 'display notification "⚡️ SSH 隧道已经在运行中，无需重复连接。" with title "VM 连接状态"'
    exit 0
fi

osascript -e 'display notification "⏳ 正在尝试连接 VM 隧道..." with title "VM 连接状态"'

for ((i=1; i<=MAX_RETRIES; i++)); do
    # 设置 10s 超时，防止死等
    ssh -o ConnectTimeout=10 -fN vm
    
    if [ $? -eq 0 ]; then
        osascript -e 'display notification "✅ 成功连接到 VM 并已建立所有端口转发！" with title "VM 连接状态"'
        exit 0
    else
        if [ $i -lt $MAX_RETRIES ]; then
            sleep $RETRY_INTERVAL
        fi
    fi
done

osascript -e 'display notification "⚠️ 连续 5 次连接 VM 失败，请稍后手动重试。" with title "VM 连接失败"'
exit 1
```

### 3.3 通过 LaunchAgent 实现开机自启动

在 `~/Library/LaunchAgents/` 目录下创建 `com.alex.vmtunnel.plist`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.alex.vmtunnel</string>
    <key>ProgramArguments</key>
    <array>
        <string>/bin/bash</string>
        <string>/Users/alex/Desktop/tools/sh/connect_vm.sh</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
</dict>
</plist>
```

**加载与测试：**
```bash
# 加载并立即触发执行
launchctl load ~/Library/LaunchAgents/com.alex.vmtunnel.plist

# 如果修改了脚本，重载即可
launchctl unload ~/Library/LaunchAgents/com.alex.vmtunnel.plist
```

## 终极工作流复盘

经历了这番调教，现在的日常变成了这样：

- **07:00** 👉 Windows PC 自动唤醒，免密码直达桌面，VMware 启动 Ubuntu，Systemd 在后台拉起所有的后端 API 和大模型服务。
- **07:10** 👉 Mac 自动唤醒。
- **09:00** 👉 我坐在工位上，输入密码进入 Mac 桌面。此时 LaunchAgent 瞬间触发重试脚本。因为 Windows 已经跑了俩小时了，脚本**一次命中**！
- 屏幕右上角弹窗：*“✅ 成功连接到 VM 并已建立所有端口转发！”* 
- 我打开浏览器输入 `127.0.0.1`，无缝衔接昨天的工作。

生命短暂，绝不把时间浪费在重复敲命令上！😎
