# uv 极简指南

很多习惯了 Node.js 生态（npm / nvm）的开发者，在刚接触 Python 时都会感到极其痛苦和混乱。Node 的 `npm install` 天生会把包放在当前目录的 `node_modules` 里；而传统的 Python `pip install` 却默认把包全装在系统全局目录中。如果不加隔离，几个项目下来，版本冲突能把系统搞崩溃（俗称“依赖地狱”）。

为了解决这个问题，Python 生态有了**“虚拟环境（venv）”**的概念（可以理解为 Python 版的 `node_modules`）。而 **`uv`** 就是目前最现代、速度最快（Rust 编写）的 Python 环境与包管理工具。

本文将为你梳理 `uv` 的核心逻辑和日常使用流程，帮你快速理清头绪。

---

## 一、 安装与基础管理

### 1. 如何安装 uv
在 macOS/Linux 终端中运行以下命令即可极速安装：
```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```
**注意：** 安装完成后，终端通常会提示你更新 `PATH` 环境变量。此时你需要刷新当前终端，让系统认识 `uv` 命令。有两个办法：
*   **做法 A：** 直接关闭当前终端窗口，重新打开一个。
*   **做法 B：** 在当前窗口执行 `source $HOME/.local/bin/env` 立即生效。

### 2. 如何查看 Python 列表（含本地自带版本）
通过以下命令可以查看当前可用的 Python 版本：
```bash
uv python list
```
**如何看懂输出结果：**
*   **`<download available>`**：表示 uv 服务器上有的版本，随时可以下载。
*   **`/usr/bin/python3.13`**：带有这种绝对路径的，表示你**操作系统自带的 Python**（比如 Debian 底层依赖的 Python）。`uv` 只是发现了它并记录在案，**它极其克制，绝对不会主动乱动你系统的原生环境**。
*   **`-> python3.13`**：这是一个软链接（相当于系统快捷方式）。说明 `python3` 这个指令实际上指向了 `python3.13`。

### 3. 如何使用 uv 安装最新稳定版的 Python
不用费劲去记各种复杂的版本号（比如带 `b` 的是测试版 Beta，纯数字的才是稳定版）。只要你敲下这个命令，`uv` 会自动为你下载当前官方主推的**最新稳定版**，并完全隔离在它自己的管理目录里：
```bash
uv python install
```
*(如果需要特定老版本兼容旧项目，也可以精确指定，例如 `uv python install 3.11`)*

## 二、 项目使用与运行机制

**核心概念：** 永远不要再尝试全局安装依赖！每一个 Python 项目，都应该有一个属于自己的 `.venv` 文件夹（也就是它的 `node_modules`）。

### 1. 新项目怎么用（极简流）
对标 Node.js，`uv` 的新项目流程极其干脆：

1. **初始化项目（类似 `npm init`）：**
   ```bash
   uv init
   ```
   *这会生成 `pyproject.toml`（等同于 package.json）和基础文件。*

2. **安装依赖包（类似 `npm install`）：**
   ```bash
   uv add requests
   ```
   *这步极其关键！`uv` 会自动在当前目录创建一个 `.venv` 虚拟环境，并把 `requests` 包安装到里面，完全不污染系统。*

3. **运行代码（类似 `node main.js`）：**
   ```bash
   uv run main.py
   ```
   *直接用 `uv run`，不需要加 `python` 这几个字。这相当于告诉系统：“去当前项目的 `.venv` 隔离房间里执行这段代码！”*

### 2. 老项目怎么处理
老项目通常没有 `pyproject.toml`，只有一个 `requirements.txt`。处理老项目时，我们可以把 `uv` 当作“速度快 100 倍的 pip”来用：

1. **进入老项目并手动创建虚拟环境房间：**
   ```bash
   uv venv
   ```
2. **极速安装老依赖：**
   ```bash
   uv pip install -r requirements.txt
   ```
   *为什么要加 `uv` 前缀？如果直接敲 `pip install`，你用的就是系统的全局 pip，会把包全装到系统里！加上 `uv`，它才会乖乖把包安装进刚建好的 `.venv` 隔离环境里。*
3. **运行老项目代码：**
   ```bash
   uv run main.py
   ```

## 三、 虚拟环境的另一种操作方式：Activate (激活模式)

前面的 `uv run main.py` 是一种**“单次委托执行”**的现代方式。
如果你觉得每次敲命令都要带上 `uv run` 太繁琐，或者你在运行一些复杂的、涉及多个脚本的老项目，你可以采用经典的**“激活模式”**。

**它的原理是：** 既然 `.venv` 是一个独立的房间，我能不能直接推门走进去？走进去之后，我就能像往常一样，直接敲 `python xxx` 了！

### 操作流程：

1. **激活虚拟环境（走入房间）：**
   ```bash
   source .venv/bin/activate
   ```
   *(执行后，你的命令行最前面通常会多出一个 `(项目名)` 或者 `(.venv)` 的标识符，提示你目前身处隔离房间中。)*

2. **在房间内自由操作：**
   进入房间后，你就不需要再带任何 `uv` 前缀了！你可以直接找回以前的老习惯：
   *   直接运行：`python main.py`
   *   直接安装依赖：`pip install requests`（放心，此时的 pip 也已经被重定向到了你的 `.venv` 里，不会弄脏系统）。

3. **退出虚拟环境（离开房间）：**
   当你在这个项目忙完了，想去弄别的项目时，只需敲入：
   ```bash
   deactivate
   ```
   *(此时提示符前的 `(项目名)` 会消失，你又回到了系统的全局环境。)*

## 四、 进阶原理解析：.venv 是像 npm 一样笨重，还是像 pnpm 一样聪明？

很多前端同学看到 `.venv` 就像看到了 `node_modules`，心里可能会犯嘀咕：*“如果在当前目录下建文件夹，那是不是每个项目都要把几十上百 MB 的依赖包重新下载、复制一遍？这岂不是和早期的 npm 一样浪费硬盘空间？”*

**答案是：过去的 pip 确实这么蠢，但现代的 `uv` 非常聪明，它的底层机制完全对标 `pnpm`！**

### 1. 告别无脑复制：引入“全局集中缓存”
`uv` 绝不会傻傻地把包重复下载。它在你的电脑底层（比如 `~/.cache/uv` 目录）建立了一个**全局缓存大仓库**。
当你第一次在一个项目中运行 `uv add requests` 时，`uv` 会把 `requests` 下载并解压到这个大仓库里。

### 2. 核心黑科技：硬链接 (Hardlinks) 与 写时复制 (CoW)
当你在第二个、第三个新项目里再次运行 `uv add requests` 时：
*   **传统 pip：** 会重新下载（或者从缓存里复制）一份完整的文件，塞进新项目的 `.venv` 里。
*   **uv 的做法：** 它会直接在这个新项目的 `.venv` 目录中，创建指向全局仓库中该包的**硬链接 (Hardlink)** 或利用现代文件系统的**写时复制 (Copy-on-Write, 例如 macOS 的 APFS 机制)**。

### 3. 这带来了什么好处？
*   **极其节省硬盘：** 哪怕你的电脑里有 100 个项目都用到了同一个版本的 `requests` 或者庞大的 `numpy`、`pandas`，它们在你的硬盘上**实际上只占用了一份空间**！
*   **极速的安装体验：** 因为只是创建了一个“链接”或者“指针”，并没有发生真实的磁盘文件读写和移动。所以当你新建一个项目并安装曾经用过的包时，安装时间几乎是 **0 毫秒**（你会看到终端提示在几毫秒内 Installed packages）。
*   **完美的隔离性：** 虽然底层指向同一个地方，但在操作系统和 Python 看来，它们就是存在于各自 `.venv` 里的独立文件，运行时的隔离机制依然完美生效。

--- 

### 💡 总结

1. **脑海里把 Python 生态和 Node 对标：**
   * `.venv` = `node_modules`
   * `pyproject.toml` = `package.json`
   * `uv add` = `npm install`
2. **别碰系统全局环境：** 让系统自带的 Python 老老实实给操作系统打工。我们要么用 `uv add`（新项目），要么用 `uv venv`（老项目）来建立专属小房间。
3. **运行代码两种流派：**
   * **懒人流：** 直接 `uv run main.py`（单次委托，推荐）。
   * **沉浸流：** 先 `source .venv/bin/activate`，然后再自由地 `python main.py`。
4. 不要害怕 `.venv` 占用空间，有了 `uv`，你就可以把它当成 **Python 界的 `pnpm`**。放心大胆地为每一个项目创建独立的 `.venv`，它既保证了项目间的绝对隔离，又榨干了性能和硬盘空间的每一分价值。
